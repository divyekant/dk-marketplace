# Delphi — Design Document

**Date**: 2026-02-27
**Status**: Draft
**Ref**: [Vision & Scope](../VISION.md)

## Architecture Decision: Approach 1 — Single Skill File

**Why**: Delphi is a prompt-driven skill. The coding agent already has every tool it needs — file reading (Glob, Grep, Read), file writing (Write), browser automation (Chrome MCP), shell execution (Bash), and web fetching. Adding an MCP server or validation scripts is premature complexity. A well-structured skill file with clear templates and explicit instructions will produce consistent output. If we hit quality issues, we graduate to a validator — but we start lean.

**Trade-off accepted**: Output consistency depends on the agent following the skill prompt precisely. We mitigate this by providing exact templates and a rigid checklist the agent must follow.

## Skill Design

### Skill Metadata

```yaml
name: delphi
description: >
  Generate and execute comprehensive test scenarios (guided cases) for any
  software. Analyzes code, docs, specs, and running apps to produce structured
  Markdown test cases covering positive, negative, edge, and accessibility
  scenarios. Can execute cases via browser automation or programmatic
  verification.
```

### Mode Detection

The skill operates in two modes, determined by how it's invoked:

| Invocation | Mode |
|------------|------|
| "Generate guided cases for X" | Generate |
| "Execute guided cases" / "Run guided cases" | Execute |
| "Generate and execute guided cases" | Generate, then Execute |
| Post-build pipeline trigger | Generate (execute if browser available) |

### Generate Mode — Process

The generate mode follows a rigid, ordered checklist:

#### Step 1: Context Gathering

The agent collects all available context:

1. **Code structure** — Glob for routes, pages, components, API handlers, CLI entry points
2. **Existing docs** — Read README, design docs, specs, API docs
3. **Recent changes** — Git log to understand what was recently built/changed
4. **Running app** (if available) — Navigate to the app, take screenshots, explore pages
5. **Existing tests** — Read test files to understand what's already covered
6. **Configuration** — Read env files, config files for feature flags, permissions

Output: Internal summary of discovered surfaces, flows, and context.

#### Step 2: Surface Discovery

From the gathered context, enumerate all testable surfaces:

- List every route/page (UI)
- List every API endpoint (API)
- List every CLI command (CLI)
- List every background job/webhook (Background)
- Group surfaces into logical flows (e.g., "Authentication", "Dashboard", "Checkout")

Output: Surface map organized by flow.

#### Step 3: Case Generation

For each flow, generate cases following this matrix:

| Dimension | Cases to Generate |
|-----------|-------------------|
| **Happy path** | The standard successful flow end-to-end |
| **Validation** | One case per validation rule (empty field, too long, wrong format, etc.) |
| **Boundary** | Min/max values, empty states, single item, many items |
| **Auth/Permissions** | Each role that should/shouldn't have access |
| **Error states** | Network failure, server error, timeout, malformed response |
| **State transitions** | What happens when you go back, refresh, navigate away mid-flow |
| **Accessibility** | Keyboard navigation, focus management, screen reader labels |
| **Concurrency** | Double-click, duplicate submission, stale data |

Each case follows the exact template from VISION.md (GC-XXX format with metadata, preconditions, steps, expected outcomes, success/failure criteria).

#### Step 4: Priority Assignment

Cases are prioritized:
- **P0**: Core happy paths + critical error handling (app crashes, data loss)
- **P1**: Validation, permissions, important edge cases
- **P2**: Cosmetic, accessibility nice-to-haves, rare edge cases

#### Step 5: Output

1. Create `tests/guided-cases/` directory structure organized by flow
2. Write each case as an individual Markdown file
3. Generate `index.md` with summary table:

```markdown
# Guided Cases Index

Generated by Delphi on YYYY-MM-DD

| ID | Title | Type | Priority | Surface | Flow | Status |
|----|-------|------|----------|---------|------|--------|
| GC-001 | Login happy path | positive | P0 | ui | auth | pending |
| GC-002 | Login invalid password | negative | P0 | ui | auth | pending |
| ... | ... | ... | ... | ... | ... | ... |

## Summary
- Total: X cases
- P0: X | P1: X | P2: X
- Positive: X | Negative: X | Edge: X | Accessibility: X
- UI: X | API: X | CLI: X | Background: X
```

### Execute Mode — Process

#### Step 1: Case Loading

1. Read `tests/guided-cases/index.md` to get the full case list
2. Accept filters: by priority, type, surface, flow, or specific case IDs
3. Parse selected case files

#### Step 2: Execution Strategy

Determine execution method per case based on its `Surface` metadata:

| Surface | Execution Method |
|---------|------------------|
| `ui` | Chrome MCP tools (navigate, find, click, read_page, screenshot) |
| `api` | Bash (curl/httpie commands) |
| `cli` | Bash (direct command execution) |
| `background` | Bash (log inspection, database queries, process checks) |

#### Step 3: Step-by-Step Execution

For each case:
1. Verify preconditions (check state, setup if needed)
2. Execute each step sequentially
3. After each step, verify the expected outcome
4. Capture evidence:
   - **UI**: Screenshot after each step
   - **API**: Full request/response
   - **CLI**: Command output
5. Record pass/fail per step
6. If a step fails, capture details and continue to next case (don't abort the run)

#### Step 4: Report Generation

Write execution report to `tests/guided-cases/reports/YYYY-MM-DD-HH-MM-report.md`:

```markdown
# Delphi Execution Report

**Run**: YYYY-MM-DD HH:MM
**Cases executed**: X of Y
**Filters applied**: [if any]

## Results Summary
- Passed: X (XX%)
- Failed: X (XX%)
- Skipped: X (XX%)

## Failures

### GC-003: Login rate limiting
- **Failed at step**: 4
- **Expected**: "Too many attempts" error after 5 failed logins
- **Actual**: No rate limiting observed after 10 attempts
- **Evidence**: [screenshot link or output]
- **Severity**: P0 — security issue

### ...

## Passed Cases
[collapsed list of passed cases]
```

## Guided Case File Format — Complete Spec

```markdown
# GC-XXX: [Descriptive Title]

## Metadata
- **Type**: positive | negative | edge | accessibility | performance | security
- **Priority**: P0 | P1 | P2
- **Surface**: ui | api | cli | background
- **Flow**: [logical flow name, e.g., "authentication", "checkout"]
- **Tags**: [comma-separated searchable tags]
- **Generated**: YYYY-MM-DD
- **Last Executed**: YYYY-MM-DD | never

## Preconditions

Bullet list of what must be true before executing this case.
Each precondition should be verifiable.

## Steps

Numbered list. Each step has:
1. **Action**: What to do (navigate, click, type, submit, call endpoint, run command)
   - **Target**: Where to do it (URL, selector description, endpoint, command)
   - **Input** (if applicable): What data to provide
   - **Expected**: What should happen (can have multiple expected outcomes)

## Success Criteria

Checklist of conditions that must ALL be true for this case to pass.

## Failure Criteria

Any ONE of these being true means the case failed.

## Notes

Optional. Known issues, flaky behavior, environment requirements.
```

## Pipeline Integration

### As a Verify-Phase Skill

Add to `pipelines.yaml`:

```yaml
skills:
  delphi:
    source: external  # or vendor/delphi
    phase: verify
    type: phase

# In the pipeline:
feature:
  phases: [explore, plan, build, verify, review, finish]
  skills:
    verify:
      - verification-before-completion
      - delphi  # generates + optionally executes after build
```

### On-Demand Invocation

Users invoke directly:
- "Generate guided cases for the auth module"
- "Execute all P0 guided cases"
- "Generate and run guided cases for what I just built"

## Phased Implementation Plan

### Phase 1: Generate Mode (Core)
**Deliverable**: `skill.md` with generate mode fully working

- Skill file with context gathering instructions
- Surface discovery logic
- Case generation with full coverage matrix
- Structured Markdown output format
- Index generation
- Sample guided case as reference

### Phase 2: Execute Mode (UI)
**Deliverable**: Execute mode for UI surface cases

- Case file parser (reads the Markdown format)
- Chrome MCP integration for step execution
- Screenshot capture per step
- Pass/fail comparison logic
- Execution report generation

### Phase 3: Execute Mode (Non-UI)
**Deliverable**: Execute mode for API, CLI, background surfaces

- Bash-based API execution (curl)
- CLI command execution
- Log/DB inspection for background cases
- Unified reporting across all surfaces

### Phase 4: Polish & Intelligence
**Deliverable**: Smarter generation and better developer experience

- Diff-aware generation (only generate cases for changed code)
- Case deduplication and pruning
- Priority scoring based on code complexity
- Integration with conductor pipeline config
- README and installation docs

|----------|----------|

| UI-only or all surfaces? | All surfaces (UI, API, CLI, background) |
| Who consumes the cases? | Both humans and AI agents |
| Pipeline or on-demand? | Both |
| Build approach? | Single skill file (Approach 1), no external deps |
